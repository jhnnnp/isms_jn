# ISMS-P 관점의 개인정보 탐지 및 비식별화 학습용 툴킷

![Version](https://img.shields.io/badge/version-0.1.0-blue)
![License](https://img.shields.io/badge/license-MIT-green)
![Python](https://img.shields.io/badge/python-3.11%2B-3776AB)
![CI](https://img.shields.io/badge/CI-GitHub_Actions-informational)

한국형 개인정보 패턴을 탐지하고, ISMS-P 보호대책 관점에서 마스킹 또는 AES-GCM 기반 암호화를 적용해 보는 Python 기반 학습용 도구입니다. 현재 구현은 CLI와 FastAPI API, 그리고 브라우저에서 바로 시연 가능한 데모 사이트를 제공하며 주민등록번호, 휴대전화번호, 이메일을 탐지합니다. 주민등록번호는 체크섬 검증을 통해 오탐을 일부 줄입니다.

이 저장소는 상용 DLP나 완전한 개인정보 탐지 솔루션을 목표로 하지 않습니다. 정형 패턴 기반 1차 필터와 비식별화 흐름을 직접 구현하고, 그 전제와 한계를 문서화하는 데 목적이 있습니다.

## 프로젝트 개요

- 목적: 로그, 텍스트 덤프, 운영 산출물에 포함된 개인정보를 사전 탐지하고 비식별화합니다.
- 지원 범위: 주민등록번호, 휴대전화번호, 이메일
- 구현 방식: Python, 정규표현식, 주민등록번호 체크섬 검증, AES-GCM 기반 암호화
- 포지셔닝: 정형 패턴 기반 1차 필터와 비식별화 흐름을 구현한 학습용 도구
- ISMS-P 연결 포인트:
  - 보호대책 `2.6`: 중요 정보 암호화 적용
  - 보호대책 `3.2`: 개인정보 비식별화 및 노출 최소화

## 데모 사이트 실행

가장 쉽게 확인하는 방법은 API 서버와 브라우저용 데모 사이트를 함께 사용하는 것입니다. 루트 페이지는 탭으로 텍스트, 파일, 복호화를 나누고, 튜토리얼, 화면과 API 대응표, 실시간 처리 흐름, 결과 요약을 함께 보여줍니다. 코드를 수정한 뒤에는 반드시 서버를 재시작해야 브라우저에 최신 데모가 반영됩니다.

```bash
source .venv/bin/activate
isms-pii-api
```

실행 후 접속 주소:

- 데모 사이트: `http://127.0.0.1:8000/`
- Swagger UI: `http://127.0.0.1:8000/docs`
- 헬스체크: `http://127.0.0.1:8000/health`

직접 `uvicorn`으로 실행해도 됩니다.

```bash
source .venv/bin/activate
uvicorn isms_pii_toolkit.api:app --host 127.0.0.1 --port 8000 --reload
```

## 사이트에서 테스트하는 방법

브라우저에서 `http://127.0.0.1:8000/`에 접속하면 다음이 보입니다.

- 상단 안내: 샘플 입력 사용법
- 텍스트 처리 탭: 단순 탐지 또는 비식별화, 암호화 대상은 칩으로 선택
- 파일 처리 탭: `text/plain` UTF-8 파일 업로드
- 복호화 탭: `ENC::...` 토큰 복호화
- 간편 튜토리얼, 화면과 API 대응표, 실시간 처리 흐름, 결과 요약

1. `텍스트 처리` 탭에서 기본 샘플 문장을 그대로 두거나 `샘플 값 초기화`로 다시 채웁니다.
2. `단순 탐지`면 `API 실행하기` 한 번으로 `POST /scan/text` 결과가 나옵니다.
3. `비식별화`를 고르면 주민등록번호 등 암호화 칩이 나타납니다. 칩으로 암호화할 유형을 고르고 키를 넣은 뒤 `API 실행하기`로 `POST /redact/text`를 호출합니다.
4. 응답이 오면 오른쪽 `실시간 처리 흐름`이 단계별로 채워집니다. (응답 수신 후 시각화)
5. `결과 요약`에서 엔드포인트와 탐지 건수를 확인합니다.
6. 비식별화가 성공하면 `복호화` 탭에 출력과 키가 채워지므로 `복호화 실행`으로 `POST /decrypt/text`를 이어갑니다.
7. `파일 처리` 탭에서 UTF-8 텍스트 파일을 올려 `POST /scan/file` 또는 `POST /redact/file`을 같은 방식으로 시연합니다.

시연 포인트:

- 탐지: 어떤 개인정보 유형이 몇 건 발견됐는지 확인
- 비식별화: 어떤 값이 마스킹됐고 어떤 값이 암호화됐는지 확인
- 복호화: `ENC::...` 토큰이 올바른 키로만 복원되는지 확인
- 과정 설명: 실행 흐름과 API 대응표로 프론트와 백엔드 연결을 설명 가능

추천 시연 입력:

```text
홍길동 900101-1234568, 010-1234-5678, hong@example.com
```

추천 키:

```text
aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa
```

## 사용법

예제 입력 파일은 `examples/sample.log`를 사용하면 됩니다.

API 중심 시연이 필요 없다면 기존 CLI도 그대로 사용할 수 있습니다.

개인정보 탐지:

```bash
isms-pii scan examples/sample.log
```

기본 마스킹:

```bash
isms-pii redact examples/sample.log
```

주민등록번호만 암호화하고 나머지는 마스킹:

```bash
export PII_TOOLKIT_AES_KEY=0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef
isms-pii redact examples/sample.log --encrypt-type rrn
```

암호화 토큰 복호화:

```bash
isms-pii decrypt redacted.log --key 0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef
```

표준 입력 사용:

```bash
echo "연락처: 010-1234-5678" | isms-pii scan
```

## 기능

현재 코드 기준 실제 구현 기능만 정리했습니다.

- 개인정보 탐지
  - 주민등록번호: `######-#######` 형식과 숫자 붙임 형식 지원
  - 휴대전화번호: `010-1234-5678`, `01012345678` 형식 지원
  - 이메일 주소 탐지
- 검증
  - 주민등록번호 체크섬 검증
  - 생년월일 형식 검증
- 비식별화
  - 마스킹: 주민등록번호, 휴대전화번호, 이메일
  - 암호화: AES-GCM 기반 암호화 토큰 생성
  - 복호화: 같은 키를 가진 호출자가 암호화 토큰 원문 복원 가능
- API 엔드포인트
  - `GET /`
  - `GET /health`
  - `POST /scan/text`
  - `POST /redact/text`
  - `POST /scan/file`
  - `POST /redact/file`
  - `POST /decrypt/text`
- 탐지 응답 메타데이터
  - 각 매치에 `validationMethod` 필드: `rrn_checksum`(체크섬·생년월일 검증) 또는 `regex_pattern`(형식 패턴만)
- 데모 사이트
  - 브라우저 기반 텍스트 탐지/비식별화 테스트
  - 텍스트 파일 업로드 기반 테스트
  - 비식별화 결과를 복호화 패널로 자동 전달
  - 튜토리얼, 화면과 API 대응, 실행 흐름, 결과 요약 제공
  - Swagger 문서와 헬스체크 링크 제공
- CLI 명령
  - `isms-pii scan`
  - `isms-pii redact`
  - `isms-pii decrypt`

## 지원 범위와 비목표

범위의 단일 소스는 `src/isms_pii_toolkit/pii_types.py`입니다.

**지원 대상**

- `rrn`: 주민등록번호
- `phone`: 휴대전화번호
- `email`: 이메일

**비목표**

- 이름·실명, 주소, 사번처럼 문맥에 의존하는 개인정보
- 계좌·카드번호, 온라인 식별자 등 현재 패턴 정의에 없는 유형
- PDF, Word, Excel, 이미지, 바이너리 로그를 직접 해석하는 범용 파일 스캐너
- 상용 DLP, OCR, 문서 분류, 데이터 거버넌스 플랫폼 대체
- 인증·권한·KMS·감사 추적·분산 레이트 리밋까지 포함한 운영용 제품

## 암호화·키·운영 방식 (구현 근거)

아래 내용은 저장소에 **실제로 구현된 코드**만을 근거로 정리한 설명입니다. 읽는 사람이 빠르게 이해할 수 있도록 먼저 쉬운 요약을 제시하고, 그 다음에 코드 기준 근거를 붙였습니다.

### 먼저 한눈에 보기

| 질문 | 짧은 답 |
| --- | --- |
| 어떤 키를 쓰는가 | 코드에 고정된 비밀키가 아니라, 실행 시 넘기는 문자열을 키 원본으로 사용합니다. |
| 어디서 키를 넣는가 | CLI는 `--key` 또는 `PII_TOOLKIT_AES_KEY`, API는 요청 본문의 `encryptionKey`를 사용합니다. |
| 어떤 알고리즘인가 | **AES-GCM**입니다. |
| 결과가 매번 같은가 | 아닙니다. 같은 평문과 같은 키라도 **nonce**가 매번 달라 결과 토큰도 달라집니다. |
| 다른 키로 풀면 어떻게 되는가 | 복호화에 실패합니다. API는 `Unable to decrypt the provided token.`만 반환합니다. |
| 전체 파일을 암호화하는가 | 아닙니다. 탐지된 개인정보 조각만 선택적으로 암호화하거나 마스킹합니다. |

### 용어부터 간단히

- **hex(16진수)**: 숫자 `0-9`와 문자 `a-f`를 사용해 데이터를 표현하는 방식입니다. 컴퓨터 안의 바이트 데이터를 사람이 비교적 짧고 읽기 쉽게 적을 때 자주 사용합니다. 예를 들어 `ff`는 16진수로 255를 뜻합니다.
- **바이트(byte)**: 컴퓨터가 데이터를 다루는 기본 단위 중 하나입니다. 보통 1바이트는 8비트이며, 암호화 키 길이를 말할 때 `16바이트`, `32바이트`처럼 표현합니다.
- **키(key)**: 암호화와 복호화에 공통으로 사용하는 비밀값입니다.
- **평문(plaintext)**: 암호화 전의 원문 문자열입니다. 예: 주민등록번호 원문
- **암호문(ciphertext)**: 암호화 후 사람이 읽을 수 없게 바뀐 데이터입니다.
- **SHA-256**: 입력 문자열을 고정 길이 결과로 바꾸는 해시 함수입니다. 이 프로젝트에서는 hex가 아닌 키 문자열을 32바이트 길이로 맞추기 위해 사용합니다.
- **Base64**: 바이너리 데이터를 문자열처럼 안전하게 전달하기 위해 문자 집합으로 바꾸는 인코딩 방식입니다.
- **URL-safe Base64**: Base64와 비슷하지만 URL이나 토큰 문자열에 넣기 쉽도록 일부 문자를 바꾼 형태입니다. 이 프로젝트는 암호화 결과를 문자열로 보여 주기 위해 이 방식을 사용합니다.
- **nonce**: 암호화할 때마다 새로 만드는 1회성 값입니다. 같은 키를 쓰더라도 결과를 매번 다르게 만드는 핵심 요소입니다.
- **tag**: 암호문이 중간에 바뀌지 않았는지 확인하는 무결성 검증 정보입니다.
- **payload**: 실제로 전달하거나 저장하는 데이터 묶음을 뜻합니다. 이 프로젝트에서는 `nonce + tag + ciphertext`를 합친 뒤 Base64로 바꾼 값을 가리킬 때 사용합니다.
- **AES**: 널리 쓰이는 대칭키 암호 알고리즘입니다. 같은 키로 암호화와 복호화를 수행합니다.
- **GCM**: AES를 사용하는 운용 방식 중 하나입니다. 암호화와 함께 무결성 검증용 tag를 만들 수 있어, 값이 바뀌었는지도 확인할 수 있습니다.
- **AEAD**: 기밀성뿐 아니라 무결성까지 함께 보장하는 암호화 방식입니다. AES-GCM이 여기에 해당합니다.

### 1. 어떤 키를 쓰는지

이 프로젝트는 저장소 안에 고정된 비밀키를 하드코딩해 두고 그것만 사용하는 방식이 아닙니다.  
암호화에 쓰이는 값은 **호출 시점에 전달되는 문자열**이며, 실행 경로에 따라 공급 방식이 다릅니다.

#### CLI에서의 키 사용 방식

- `src/isms_pii_toolkit/cli.py`의 `_resolve_key()`는 `--key`가 있으면 그것을 우선 사용합니다.
- `--key`가 없으면 환경 변수 `PII_TOOLKIT_AES_KEY`를 사용합니다.
- `decrypt` 명령에서 둘 다 없으면 실행 자체가 실패합니다.

즉 CLI 사용자는 둘 중 하나로 키를 넣으면 됩니다.

```bash
isms-pii redact examples/sample.log --encrypt-type rrn --key <키 문자열>
```

또는

```bash
export PII_TOOLKIT_AES_KEY=<키 문자열>
isms-pii redact examples/sample.log --encrypt-type rrn
```

#### API에서의 키 사용 방식

- API는 환경 변수 `PII_TOOLKIT_AES_KEY`를 자동으로 읽어 암호화에 쓰지 않습니다.
- `src/isms_pii_toolkit/schemas.py`의 요청 모델에서 `encryptionKey` 필드를 받고,
- `src/isms_pii_toolkit/api.py`의 `_ensure_encryption_key()`가 `encryptTypes`가 있을 때 이 값이 비어 있으면 400 에러를 발생시킵니다.

즉 API는 아래처럼 **요청 본문에 키를 직접 담아 보내는 구조**입니다.

```json
{
  "text": "홍길동 900101-1234568",
  "encryptTypes": ["rrn"],
  "encryptionKey": "aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa"
}
```

#### 예시 키에 대한 해석

`.env.example`에 있는 값은 다음과 같습니다.

```env
PII_TOOLKIT_AES_KEY=0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef
```

이 값은 **실운영 비밀키가 아니라 예시용 샘플**입니다.  
문서와 데모를 설명하기 위한 값일 뿐, 실제 배포 환경에서 그대로 사용하면 안 됩니다.

#### `aaaaaaaa...` 같은 예시 키만 꼭 써야 하는가

아닙니다. 문서나 테스트에서 자주 보이는 `aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa` 같은 값은 **설명을 쉽게 하기 위한 예시 키**일 뿐입니다. 이 프로젝트는 특정 문자열 하나만 허용하지 않습니다.

다음과 같은 값들도 모두 입력 가능한 키가 될 수 있습니다.

- 64자리 hex 문자열 예: `bbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbb`
- 다른 64자리 hex 문자열 예: `0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef`
- 일반 문자열 예: `my-secret-key`

중요한 점은 “무슨 문자열을 넣느냐”보다, **암호화할 때 사용한 키와 복호화할 때 사용하는 키가 같아야 한다**는 것입니다.

### 2. 키 문자열이 실제 AES 키로 바뀌는 방식

AES는 아무 문자열이나 그대로 키로 받을 수 없습니다.  
키 길이가 **16바이트, 24바이트, 32바이트** 중 하나여야 합니다. 이 프로젝트는 이 제약을 맞추기 위해 `src/isms_pii_toolkit/crypto.py`의 `_normalize_key()`를 사용합니다.

동작 순서는 다음과 같습니다.

1. 먼저 입력 문자열을 `bytes.fromhex()`로 해석해 봅니다.
2. 성공하면 “16진 문자열로 표현된 바이트 키”라고 보고 그대로 사용합니다.
3. 실패하면 일반 문자열이라고 보고 UTF-8로 인코딩한 뒤 SHA-256을 한 번 적용해 32바이트를 만듭니다.
4. 결과 길이가 16, 24, 32바이트가 아니면 예외를 발생시킵니다.

#### 예시 1. 64자리 hex 문자열

- 입력: `0123456789abcdef...` 같은 64자리 hex 문자열
- 처리: 2 hex 문자 = 1바이트이므로 총 32바이트
- 결과: **AES-256 길이의 키**

#### 예시 2. 일반 문장 형태의 문자열

- 입력: `my-secret-key`
- 처리: hex 해석 실패 -> SHA-256 적용
- 결과: 32바이트
- 의미: **AES-256 길이로 정규화된 키**

#### 서로 다른 키를 쓰면 어떻게 되는가

키를 이해할 때 가장 중요한 동작은 아래 두 경우입니다.

1. **암호화와 복호화에 같은 키를 쓰는 경우**
   - 정상 동작합니다.
   - 예를 들어 키 A로 주민등록번호를 암호화하고, 나중에 다시 키 A로 복호화하면 원문이 복원됩니다.

2. **암호화와 복호화에 다른 키를 쓰는 경우**
   - 복호화에 실패합니다.
   - 예를 들어 키 A로 암호화한 뒤 키 B로 복호화하면 `decrypt_and_verify()` 단계에서 인증 태그 검증이 통과하지 못합니다.
   - API에서는 이 경우 구체 사유를 세분화하지 않고 `Unable to decrypt the provided token.`만 반환합니다.

쉬운 말로 바꾸면, 이 프로젝트는 “어떤 키든 쓸 수 있지만, **한 번 잠근 열쇠로 다시 열려면 같은 열쇠가 필요하다**”는 구조입니다.

#### 왜 이렇게 만들었는가

사용자 입장에서는 hex 문자열이든 일반 문자열이든 넣을 수 있어야 CLI와 데모 사용이 편합니다.  
다만 보안적으로는 중요한 주의점이 있습니다.

- 일반 문자열 입력에 대한 SHA-256 처리는 **편의상 길이를 맞추는 정규화**입니다.
- 이것은 **비밀번호 전용 KDF(PBKDF2, scrypt, Argon2)** 와는 다릅니다.
- 따라서 운영 환경에서는 추측하기 쉬운 문자열보다 **충분히 긴 무작위 키**를 hex 형태로 쓰는 것이 더 안전합니다.

### 3. 어떤 암호화 방식을 쓰는지

이 프로젝트는 **AES-GCM**을 사용합니다.

- **AES**: 널리 사용되는 대칭키 블록 암호
- **GCM 모드**: 암호화와 함께 무결성 검증용 태그를 생성하는 운용 방식

즉 이 프로젝트는 “값을 숨기는 것”뿐 아니라 “중간에 바뀌지 않았는지 확인하는 것”까지 함께 처리합니다.

#### 코드상 실제 처리 방식

`src/isms_pii_toolkit/crypto.py`의 `encrypt_text()`는 다음 순서로 동작합니다.

1. 전달된 키 문자열을 `_normalize_key()`로 정규화
2. `get_random_bytes(12)`로 12바이트 nonce 생성
3. `AES.new(raw_key, AES.MODE_GCM, nonce=nonce)`로 GCM cipher 생성
4. `encrypt_and_digest()`로 암호문과 tag 생성
5. `nonce + tag + ciphertext`를 하나로 붙임
6. 전체를 `base64.urlsafe_b64encode()`로 문자열화

복호화 함수 `decrypt_text()`는 반대로 다음 순서로 처리합니다.

1. Base64 문자열을 디코딩
2. 앞 12바이트를 nonce로 추출
3. 그다음 16바이트를 tag로 추출
4. 나머지를 ciphertext로 추출
5. 같은 키로 `decrypt_and_verify()` 수행
6. 태그 검증까지 통과하면 UTF-8 평문으로 복원

#### 저장되는 실제 토큰 형태

암호화된 값은 본문 안에서 다음 형태로 들어갑니다.

```text
ENC::<pii_type>::<base64 payload>
```

예를 들어 주민등록번호가 암호화되면 대략 이런 꼴입니다.

```text
ENC::rrn::<암호화된 Base64 문자열>
```

이 형식은 `src/isms_pii_toolkit/redactor.py`에서 만들어지며, 복호화할 때도 같은 파일의 `ENCRYPTED_TOKEN_PATTERN`이 이 패턴을 찾아 payload 부분만 꺼내 처리합니다.

### 4. 왜 "비식별화"라는 표현을 쓰는지

이 프로젝트에서 `비식별화`라는 표현을 쓰는 이유는, 개인정보를 무조건 삭제하거나 완전히 익명화하는 것이 아니라 **식별 가능성을 낮추는 여러 처리 방식**을 함께 사용하기 때문입니다.

현재 구현 기준으로 보면 `src/isms_pii_toolkit/redactor.py`의 `redact_text()`는 탐지된 개인정보마다 아래 두 전략 중 하나를 선택합니다.

- `mask`: 일부를 가려 즉시 식별하기 어렵게 만듭니다.
- `encrypt`: 원문을 `ENC::...` 토큰으로 바꾸어 바로 읽을 수 없게 만듭니다.

즉 이 프로젝트는 단순히 “지운다”기보다, **개인정보 노출을 줄이고 식별 가능성을 낮추는 처리**를 수행합니다. 그래서 문서에서 `마스킹`, `암호화`, `복호화`를 각각 따로 설명하되, 이를 묶는 상위 개념으로 `비식별화`라는 표현을 사용합니다.

조금 더 정확히 말하면 다음과 같습니다.

1. **마스킹은 비식별화의 한 형태입니다.**
   - 주민등록번호, 전화번호, 이메일 일부를 가려서 원문을 바로 알아보기 어렵게 만듭니다.

2. **암호화도 이 프로젝트에서는 비식별화 흐름의 일부로 사용됩니다.**
   - 값이 `ENC::...` 토큰으로 바뀌어 평문이 직접 노출되지 않습니다.
   - 다만 같은 키가 있으면 다시 복호화할 수 있으므로, “영구적으로 복원 불가능한 익명화”와는 다릅니다.

3. **그래서 "익명화"보다 "비식별화"가 더 정확합니다.**
   - 완전한 익명화는 일반적으로 원문 복원이 불가능해야 하지만,
   - 이 프로젝트의 암호화 결과는 적절한 키가 있으면 다시 복원할 수 있습니다.
   - 따라서 현재 구현을 가장 무리 없이 설명하는 표현은 `비식별화`입니다.

이 표현은 API의 컴플라이언스 메시지와도 맞습니다.

- `src/isms_pii_toolkit/api.py`의 `COMPLIANCE_NOTES`는 `마스킹 및 암호화 결과를 분리해 최소 권한 기반 처리 흐름을 지원`한다고 설명합니다.
- README 프로젝트 개요도 `개인정보를 사전 탐지하고 비식별화`한다고 정의합니다.

정리하면, 이 프로젝트에서 `비식별화`는 **마스킹과 선택적 암호화를 포함하는 상위 개념**으로 쓰이고 있으며, 구현 범위와 가장 잘 맞는 표현입니다.

### 5. 왜 AES-GCM을 사용했는지

이 프로젝트의 목적은 README 앞부분에 적혀 있듯이, ISMS-P 관점에서 **개인정보 비식별화와 암호화 흐름을 학습·시연**하는 것입니다.  
그 목적에 비추어 보면 AES-GCM을 택한 이유는 다음처럼 설명할 수 있습니다.

#### 1. 기밀성과 무결성을 함께 다룰 수 있기 때문

단순 암호화만 쓰면 “읽을 수는 없지만 중간에 변조되었는지”를 별도로 확인해야 할 수 있습니다.  
AES-GCM은 tag를 통해 복호화 시 무결성까지 검증하므로, 학습용 도구에서도 더 완결된 보호 흐름을 보여 줄 수 있습니다.

#### 2. 선택적 필드 보호와 잘 맞기 때문

이 프로젝트는 문서 전체를 통째로 암호화하지 않고, 주민등록번호·전화번호·이메일 같은 **부분 문자열**만 골라 처리합니다.  
AES-GCM은 이런 짧은 텍스트 조각을 보호하고, 나중에 원문 복원이 필요한 시나리오를 설명하기에 적합합니다.

#### 3. 현재 구현 구조가 "같은 키로 다시 복호화"를 전제로 하기 때문

이 프로젝트의 현재 코드 흐름은 `encrypt_text(value, key)`와 `decrypt_text(token, key)`처럼 **같은 종류의 키 입력을 다시 받아 복호화하는 구조**입니다.  
즉 공개키와 개인키를 나누어 관리하는 구조가 아니라, 하나의 비밀값을 공유해 암호화와 복호화를 수행하는 **대칭키 모델**에 맞춰 설계되어 있습니다.

실제로 구현된 코드에는 다음과 같은 특징이 있습니다.

- `src/isms_pii_toolkit/crypto.py`는 단일 `key: str` 입력만 받아 `_normalize_key()` 후 AES 키로 사용합니다.
- `src/isms_pii_toolkit/cli.py`는 `--key` 또는 `PII_TOOLKIT_AES_KEY` 하나만 받습니다.
- `src/isms_pii_toolkit/api.py`와 `src/isms_pii_toolkit/schemas.py`도 `encryptionKey` 하나만 요청받습니다.
- 공개키/개인키 쌍, 인증서, 키 식별자, 키 교환 절차를 다루는 구현은 현재 저장소에 없습니다.

즉 이 저장소의 실제 구현은 “선택한 개인정보 조각을 암호화해 토큰으로 바꾸고, 나중에 **같은 비밀키를 가진 호출자**가 다시 원문을 복원한다”는 흐름에 정확히 맞춰져 있습니다.

#### 왜 비대칭키를 쓰지 않았는가

비대칭키 암호화는 일반적으로 **공개키로 암호화하고 개인키로 복호화**하는 구조를 말합니다. 이 방식은 수신자 분리, 키 배포, 인증서 기반 신뢰 체계 같은 시나리오에 강점이 있지만, 현재 이 프로젝트의 범위와 구현 방식에는 다음 이유로 맞지 않습니다.

1. **현재 사용 시나리오가 "같은 주체가 암호화 후 다시 복호화"하는 흐름이기 때문**
   - 이 프로젝트는 CLI 사용자나 API 호출자가 같은 키를 넣어 암호화하고, 필요할 때 다시 같은 키로 복호화하는 구조입니다.
   - “누군가의 공개키로 잠그고, 그 사람의 개인키만으로 푼다”는 수신자 분리 모델이 현재 요구사항에 직접 등장하지 않습니다.

2. **구현 복잡도를 불필요하게 늘리기 때문**
   - 비대칭키를 도입하면 공개키/개인키 생성, 배포, 저장, 교체, 유효성 관리 같은 별도 설계가 필요합니다.
   - 현재 저장소는 학습용·데모용 툴킷이며, README의 비목표에도 KMS/HSM, 키 회전, 운영용 키 관리 체계가 빠져 있습니다.

3. **현재 보호 대상이 짧은 텍스트 조각이기 때문**
   - 이 프로젝트는 파일 전체를 잠그는 것이 아니라 주민등록번호·전화번호·이메일 같은 **부분 문자열**을 선택적으로 처리합니다.
   - 이런 구조에서는 현재처럼 간단한 토큰 기반 대칭키 암호화가 설명과 시연 모두에 더 직관적입니다.

4. **실제 코드에 비대칭키 관련 구현이 없기 때문**
   - RSA, ECC, 공개키 PEM 로딩, 개인키 복호화, 인증서 검증 같은 구현은 현재 저장소에 없습니다.
   - 따라서 문서에서도 “비대칭키를 사용한다” 또는 “비대칭키를 운영적으로 지원한다”라고 쓰면 실제 구현과 어긋납니다.

정리하면, 이 프로젝트는 **수신자 분리형 공개키 암호화 시스템**이 아니라, **같은 호출자가 선택적 암호화와 복호화를 시연하는 대칭키 기반 도구**이기 때문에 AES-GCM이 현재 구현과 더 잘 맞습니다.

#### 다른 대칭키 방식들은 왜 쓰지 않았는가

현재 저장소는 **AES-GCM만 구현하고 테스트**하고 있습니다. 아래 비교는 “다른 방식이 절대 나쁘다”는 뜻이 아니라, **현재 프로젝트의 목적과 구현 범위에서 왜 채택하지 않았는지**를 설명하기 위한 것입니다.

| 방식 | 특징 | 현재 프로젝트에서 채택하지 않은 이유 |
| --- | --- | --- |
| `AES-ECB` | 가장 단순한 블록 모드 중 하나 | 같은 패턴이 반복될 때 정보가 드러날 수 있어 민감정보 보호 예시로 부적절합니다. 현재 저장소에도 구현이 없습니다. |
| `AES-CBC` | 오래전부터 널리 알려진 블록 모드 | 암호화는 가능하지만 무결성 검증이 내장되지 않아, 현재처럼 `decrypt_and_verify()` 한 번으로 설명되는 구조보다 복잡해집니다. 별도 MAC 설계도 필요합니다. |
| `AES-CTR` | 스트림처럼 동작하는 대칭키 방식 | 빠르고 유연할 수 있지만 CBC와 마찬가지로 무결성 보호가 내장되어 있지 않아, 현재 프로젝트의 “암호화 + 변조 검증” 설명과 바로 맞지 않습니다. |
| `ChaCha20-Poly1305` | 현대적인 AEAD 방식 | 충분히 유효한 대안이지만, 현재 저장소는 AES 중심으로 구현·문서화되어 있고 `src/isms_pii_toolkit/crypto.py`도 AES-GCM만 사용합니다. 학습용 범위를 넓히기보다 하나의 표준 구현에 집중한 것입니다. |

즉 현재 프로젝트는 다음 우선순위를 택한 셈입니다.

1. **기밀성과 무결성을 한 번에 설명할 수 있을 것**
2. **부분 문자열 토큰화 구조와 잘 맞을 것**
3. **코드와 README를 처음 보는 사람도 이해하기 쉬울 것**
4. **실제 저장소에 구현과 테스트가 존재할 것**

이 기준에서 가장 설명이 단순하고 현재 구현과 일치하는 선택이 **AES-GCM**입니다.

참고로 GCM의 성능·병렬화 장점은 일반적으로 알려져 있지만, **이 저장소는 성능 벤치마크 결과를 근거로 선택했다고 문서화한 상태는 아닙니다.** 따라서 README에서는 “표준적이고 무결성 검증이 가능한 방식”이라는 수준의 설명이 가장 정확합니다.

### 6. 실제 운영 흐름: 언제 암호화하고 언제 마스킹하는지

이 프로젝트는 텍스트 전체를 암호화하지 않습니다.  
먼저 `detect_pii()`로 개인정보를 찾고, 그 다음 `redact_text()`에서 **각 매치마다 전략을 선택**합니다.

전략은 두 가지입니다.

- `mask`: 원문 일부를 가리는 방식
- `encrypt`: 원문을 `ENC::...` 토큰으로 치환하는 방식

#### 동작 예시

입력이 아래와 같다고 가정합니다.

```text
홍길동 900101-1234568, 010-1234-5678, hong@example.com
```

그리고 암호화 대상으로 `rrn`만 지정하면,

- 주민등록번호: 암호화
- 전화번호: 마스킹
- 이메일: 마스킹

처럼 처리됩니다.

즉 결과는 개념적으로 아래와 비슷해집니다.

```text
홍길동 ENC::rrn::<payload>, 010-****-5678, h***@example.com
```

이 구조 덕분에 한 문장 안에서도 “어떤 값은 되돌릴 수 있게 암호화하고, 어떤 값은 복구 불가능한 마스킹으로만 처리”하는 흐름을 동시에 시연할 수 있습니다.

### 7. 왜 암호화할 때마다 결과가 바뀌는지

같은 주민등록번호를 같은 키로 두 번 암호화하면, 사람 입장에서는 같은 결과가 나와야 할 것처럼 느껴질 수 있습니다.  
하지만 이 프로젝트의 결과 토큰은 **매번 달라지는 것이 정상**입니다.

이유는 `encrypt_text()` 안의 이 한 줄 때문입니다.

- `nonce = get_random_bytes(12)`

즉 암호화할 때마다 새 nonce를 랜덤하게 생성합니다.  
그 결과 다음이 성립합니다.

- 평문이 같아도
- 키가 같아도
- nonce가 다르면
- 최종 Base64 토큰이 달라집니다

#### 왜 일부러 다르게 만드는가

같은 값이 항상 같은 암호문으로 바뀌면, 공격자가 암호문 패턴만 보고 “이 둘은 같은 원문이구나” 같은 추론을 하기 쉬워집니다.  
랜덤 nonce는 이런 반복 패턴 노출을 줄여 줍니다.

#### 중요한 점

- 결과 문자열은 매번 달라질 수 있습니다.
- 하지만 복호화에 필요한 nonce와 tag가 payload 안에 같이 들어 있으므로,
- **같은 키만 맞으면 원문은 정확히 복원됩니다.**

즉 “암호문은 매번 다르지만, 평문은 같은 키로 정상 복호화된다”가 핵심입니다.

### 8. 다른 키를 쓰면 어떻게 되는지

암호화할 때 쓴 키와 복호화할 때 쓰는 키는, 내부적으로 정규화된 결과까지 포함해 **동일해야 합니다.**  
다른 키를 사용하면 `decrypt_and_verify()` 단계에서 실패합니다.

실패 원인은 대표적으로 아래와 같습니다.

- 복호화 키가 다름
- 토큰이 일부 잘림
- Base64 문자열이 손상됨
- payload 안의 nonce, tag, ciphertext 일부가 변조됨

이 경우 `src/isms_pii_toolkit/crypto.py`에서는 `ValueError("Invalid encrypted token.")`가 발생합니다.  
그리고 API 레이어에서는 이 내부 사유를 그대로 노출하지 않고 `Unable to decrypt the provided token.`으로 통일합니다.

#### 왜 오류를 일반화하는가

클라이언트에게 “키가 틀렸다”, “토큰이 깨졌다”, “태그가 맞지 않는다” 같은 세부 사유를 과하게 노출할 필요는 없습니다.  
이 프로젝트는 외부 응답에서는 일반화된 실패 메시지만 내려주고, 내부 구현에서만 상세 예외를 처리하는 방식을 택하고 있습니다.

#### 테스트로도 확인되는가

네. `tests/test_api.py`에는 다음 시나리오가 있습니다.

- 같은 키로 암호화 후 같은 키로 복호화 -> 성공
- 깨진 토큰으로 복호화 -> 400 실패
- 다른 키로 복호화 -> 400 실패

즉 문서 설명이 아니라, 실제 테스트로도 동작이 검증되어 있습니다.

### 9. 한 번에 이해하는 전체 처리 흐름

아래는 사용자가 텍스트 비식별화를 요청했을 때 실제로 일어나는 순서를 쉬운 흐름으로 풀어쓴 것입니다.

1. 입력 텍스트를 받습니다.
2. `detect_pii()`가 주민등록번호·전화번호·이메일을 찾습니다.
3. `encryptTypes`에 포함된 유형인지 확인합니다.
4. 포함된 유형이면 `encrypt_text()`로 AES-GCM 암호화를 수행합니다.
5. 포함되지 않은 유형이면 `mask_value()`로 마스킹합니다.
6. 결과 문자열을 합쳐 최종 출력 본문을 만듭니다.
7. 나중에 사용자가 복호화를 요청하면 `ENC::...` 토큰만 찾아 `decrypt_text()`를 적용합니다.
8. 키가 맞고 토큰이 정상적이면 원문을 복원하고, 아니면 실패합니다.
